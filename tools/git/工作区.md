## 工作区（Working Tree）是什么

**工作区**就是你项目目录里**那些真实存在、你正在编辑的文件**（`.md`、`.cpp`、`.py`、图片、配置等）。  
你在 VS Code/终端/任何编辑器里改的内容，**首先都发生在工作区**。

> 一句话：**工作区 = 你手里正在写的“草稿纸”**。


## 工作区里常见的“文件状态”

用 `git status` 你会看到类似这些状态，它们都发生在工作区层面：

### 1) Untracked（未跟踪）

- 文件存在于工作区，但 Git 还不知道它（没 add 过）
    
- 典型：新建了 `notes.md`

### 2) Modified（已修改）

- 文件已被 Git 跟踪过，但你又改了它（改动在工作区）
    
- 典型：你改了 `main.cpp`

### 3) Deleted（已删除）

- 你把一个被跟踪的文件删了（工作区发生删除）
    
 ### 备注：`git add -A` 会把新增/修改/删除全部纳入暂存区，非常常用。


## 工作区的 3 个关键命令

### 1) 看当前工作区发生了什么

`git status`

你应该重点看两块：

- **Changes not staged for commit**：工作区改了，但还没 add
    
- **Untracked files**：工作区新文件，还没被 Git 管
    

---

### 2) 看工作区具体改了哪些行

`git diff`

> `git diff` 默认是：**工作区 vs 暂存区** 的差异  
> （也可以理解成：你改了什么但还没 add）

---

### 3) 丢弃工作区改动（危险操作）

把工作区某文件恢复到“上一次提交/或暂存区”的样子：

`git restore file.txt`

丢弃所有工作区改动：

`git restore .`

⚠️ 注意：

- 这会直接丢掉你在工作区的改动（除非你有别的备份）
    
- 丢之前最好先 `git diff` 看清楚
